# 一、文档原文关于准确率的全部规定
1. 量化指标：**答案准确率≥85%**，判定标准为**人工校验**。
2. 业务口径：针对金融制度问答场景，答案需匹配对应原始制度条文，且要求**答案溯源**（标注对应制度条文）。
3. 配套约束：仅使用金融监管官网公开数据/企业公开版文档作为数据源。

# 二、当前准确率验证环节 缺失项（对照你提出的点）
1. **无标准测试集（Ground-Truth）**
   缺少 `(查询问句, 标准答案/对应法规条文)` 标注样本对，无法开展批量校验。
2. **无统一指标定义**
   文档只提“准确率≥85%、人工校验”，未界定**“准确”的判定规则**。
3. **无自动化评估脚本**
   仅靠人工校验效率低，无批量跑测、指标计算、结果统计工具。
4. **无标准化评估报告**
   缺少分维度数据统计（检索命中率、答案正确率、溯源准确率等）。

# 三、结合项目要求 明确核心规则（对齐文档+落地执行）
## 1. 准确率定义（优先贴合原文「人工校验+条文溯源」要求）
采用**双维度判定**，整体准确率综合计算，满足才算有效：
- 维度1（检索层）：RAG 检索返回的 Top-K 条文**包含真实对应法规条文**（溯源命中）；
- 维度2（生成层）：LLM 输出答案**语义、内容、条款描述与标准答案一致**，无错误解读、无编造内容；
- 判定规则：**两条同时满足，单条样本记为“准确”**，否则为“错误”；
- 不采用纯字符完全匹配（金融问答允许句式改写），以**语义+条文匹配**为核心。

## 2. 测试集要求
### （1）样本数量
结合项目要求「1000+ 核心知识点」，建议测试集规模：**不少于200条标注样本**（覆盖不同法规、不同提问类型：关键词查询、自然语言问答、关联查询）。
### （2）样本来源（二选一，推荐方案1）
1. **推荐**：基于项目内已导入的公开金融法规文档，通过脚本自动生成问句 + 人工修正标注，生成 `(Query, 标准条文ID/标准答案)` 标准数据集；
2. 备选：直接使用企业提供的测试用例模板，人工整理历史业务问句。
### （3）样本格式（Ground-Truth 标准格式）
```
{
  "query": "用户提问语句",
  "target_doc_id": "对应法规条文唯一ID（用于溯源校验）",
  "target_content": "标准参考答案/原始法规条文"
}
```

## 3. 评估方式（结合文档「人工校验」要求，采用「半自动评估」）
兼顾落地效率与项目规则，分两层执行：
1. **第一层：自动化初筛**
   - 规则校验：比对检索结果中的条文ID 是否命中 `target_doc_id`（溯源校验）；
   - 语义校验：使用开源LLM（通义千问/ChatGLM，项目指定模型）作为裁判模型，判断系统答案与标准内容语义是否一致；
2. **第二层：人工复核（必做，对齐文档要求）**
   对自动化判定结果抽样/全量人工校验，最终**以人工结果作为最终准确率依据**。
> 禁止使用文档外的第三方模型（如GPT-4），统一使用项目指定开源可商用LLM。

## 4. 评估指标体系
除整体准确率外，拆分维度统计，输出完整评估报告：
1. **整体答案准确率** = 准确样本数 / 总测试样本数 （目标 ≥85%）
2. **检索命中率**：检索阶段Top-K 命中目标法规条文的样本占比
3. **溯源准确率**：答案中标注的溯源条文与真实条文一致的样本占比
4. 附加统计：平均响应耗时（校验 ≤2秒 的性能要求）

## 5. 必备工程文件（补齐缺失工具）
1. **标准测试集文件**：`test_gt.json`（存放所有标注好的 Query + Ground-Truth）
2. **自动化评估脚本**
   - 功能：批量调用问答接口、获取检索结果+答案、执行规则/语义校验、计算各项指标、生成日志；
   - 技术：基于Python + Postman接口逻辑对接后端，输出结构化统计结果。
3. **评估报告模板**
   包含：测试样本总量、各维度指标数值、错误案例汇总、响应耗时统计、人工复核说明。
